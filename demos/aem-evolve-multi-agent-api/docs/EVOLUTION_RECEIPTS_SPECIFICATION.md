## 1. Overview

An **Evolution Receipt** is the formal, verifiable record generated by the **AEM-EVOLVE™** governance system after evaluating a proposed material change, known as an **Evolution Event**, in an AI system.

Evolution Receipts record governance decisions, preserve event-to-decision traceability, enforce explicit scope boundaries, and support auditability of changes in agentic AI systems.

In this technical demonstration, Evolution Receipts provide hash-based integrity evidence through canonical SHA-256 hashing. In signed implementations, receipts may also provide cryptographic signature verification.

---

## 2. Purpose

Evolution Receipts fulfill the following roles:

- Record the outcome of an Evolution Gate: `PASS`, `SCOPE_LIMITED`, `FAIL_CLOSED`, `REJECTED_OVERCLAIM`, etc.
- Provide reasoning and justification for the governance decision.
- Preserve linkage to the originating Evolution Event.
- Enforce explicit **claim boundaries**.
- Support public verification and audit inspection.
- Support Human-in-the-Loop approval workflows.

---

## 3. Schema

All Evolution Receipts in this technical demonstration conform to the following JSON structure:

```json
{
  "schema_id": "AEM_EVOLVE_EVOLUTION_RECEIPT_SCHEMA_V1",
  "receipt_id": "string",
  "receipt_payload": {},
  "event_id": "string",
  "thread_id": "string",
  "event_canonical_sha256": "string",
  "receipt_canonical_sha256": "string",
  "signature_status": "string",
  "timestamp_utc": "string"
}
```

## 4. Field Descriptions

| Field | Type | Required | Description |
|---|---:|---:|---|
| `schema_id` | string | Yes | Fixed identifier: `AEM_EVOLVE_EVOLUTION_RECEIPT_SCHEMA_V1` |
| `receipt_id` | string | Yes | Unique receipt identifier, typically `REC-{event_id}` |
| `receipt_payload` | object | Yes | Core decision and constraint data |
| `event_id` | string | Yes | Reference to the originating Evolution Event |
| `thread_id` | string | Yes | Workflow session identifier |
| `event_canonical_sha256` | string | Yes | SHA-256 hash of the associated Evolution Event |
| `receipt_canonical_sha256` | string | Yes | SHA-256 hash of the canonical receipt object before this hash field is attached |
| `signature_status` | string | Yes | Signature state, such as `NOT_SIGNED_DEMO` or an algorithm name |
| `timestamp_utc` | string | Yes | ISO 8601 creation timestamp |

## 5. Receipt Payload Structure

The `receipt_payload` contains the actual governance decision:

```json
{
  "receipt_payload": {
    "outcome": "SCOPE_LIMITED",
    "receipt_message": "Cambio aprobado con limitaciones de scope.",
    "materiality_score": 78.0,
    "claim_boundary": {
      "research_support_only": true,
      "clinical_claimed": false,
      "diagnostic_claimed": false,
      "regulatory_approval_claimed": false,
      "third_party_binding": false
    },
    "requested_claim_scope": "RESEARCH_SUPPORT"
  }
}
```

### Fields

| Field | Type | Description |
|---|---:|---|
| `outcome` | string | Gate decision: `PASS`, `SCOPE_LIMITED`, `FAIL_CLOSED`, `REJECTED_OVERCLAIM`, etc. |
| `receipt_message` | string | Human-readable explanation of the decision |
| `materiality_score` | number | Change impact score, typically 0–100 |
| `claim_boundary` | object | Explicit limits on allowed claims |
| `requested_claim_scope` | string | Originally requested scope |

## 6. Claim Boundary Object

The `claim_boundary` explicitly defines what claims are not allowed after the change:

```json
{
  "claim_boundary": {
    "research_support_only": true,
    "clinical_claimed": false,
    "diagnostic_claimed": false,
    "regulatory_approval_claimed": false,
    "third_party_binding": false
  }
}
```

This mechanism is central to **Mechanical Ethics** because it prevents systems from making claims beyond their verified capabilities.

A receipt may allow a limited technical or research-support outcome while explicitly preserving boundaries against clinical, diagnostic, regulatory, or third-party-binding claims.

## 7. Integrity and Hashing

Every receipt includes two integrity fields:

- `event_canonical_sha256`: hash of the original Evolution Event.
- `receipt_canonical_sha256`: hash of the canonical receipt object before this hash field is attached.

The canonical hash is calculated using JSON canonicalization with:

```python
json.dumps(data, sort_keys=True, separators=(",", ":"), ensure_ascii=False)
```

Then SHA-256 is applied to the resulting UTF-8 encoded canonical JSON.

These hashes allow independent verification of both the event and the receipt. The hash fields themselves are excluded from their respective preimage to avoid self-referential hashing.

---

## 8. Signature Status

| Value | Meaning |
|---|---|
| `NOT_SIGNED_DEMO` | Receipt is not cryptographically signed; demonstration mode only |
| `RSA-PSS-SHA256` | Receipt was signed using RSA-PSS with SHA-256 |
| Other algorithm name | Implementation-specific signature scheme |

In this technical demonstration, receipts use:

```text
signature_status = NOT_SIGNED_DEMO
```

Production or higher-assurance implementations should use cryptographic signatures and signature verification records.

## 9. Relationship with Evolution Events

```text
Evolution Event
    ↓
Evolution Gate
    ↓
Evolution Receipt
```

The **Evolution Event** describes what change is proposed.

The **Evolution Gate** evaluates the event against materiality, scope, evidence, and claim boundaries.

The **Evolution Receipt** records what was decided and under what constraints.

The receipt links back to the event through:

- `event_id`
- `event_canonical_sha256`
- `thread_id`

## 10. Usage in the Technical Demonstration

In this demonstration:

- Every agent-proposed change generates an Evolution Event.
- The Evolution Gate evaluates the event and produces an Evolution Receipt.
- Receipts are stored in the `evolution_receipts` table.
- Events are stored in the `evolution_events` table.
- Human decisions are stored in the `human_decisions` table.
- When the outcome is `SCOPE_LIMITED`, the workflow pauses for human approval.
- Receipts are queryable through `/receipt/{thread_id}` and `/audit/{thread_id}`.
- Events are queryable through `/event/{thread_id}`.
- The audit endpoint returns events, receipts, and human decisions.
