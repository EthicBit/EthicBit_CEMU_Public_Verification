## 3. Historical Architecture Layers

### Layer 1 — EthicBit Doctrinal Framework (Mechanical Ethics)

The EthicBit layer defines the normative doctrine layer for AI system governance. It establishes the Mechanical Ethics standard: formal, verifiable, and scope-bounded behavioral constraints for AI operations.

- Defines what constitutes ethical mechanical operation
- Establishes the claim boundary discipline applied throughout the stack
- Governs which assertions may and may not be made at each operational layer

### Layer 2 — CEERV Standard (Offline Verifiable Evidence)

CEERV (Controlled Evidence of Ethical Runtime Verification) defines the offline verifiable evidence layer.

- Produces cryptographic evidence packages that can be verified without network access
- Establishes SHA-256 manifest integrity references
- Provides the evidence-chain that connects runtime behavior to declared claims
- Evidence packages are bound by declared scope, not asserted universally

### Layer 3 — CEMU Capsule (Operational Enforcement)

CEMU (Controlled Ethical Mechanical Unit) is the operational capsule that executes, seals, verifies, and governs the end-to-end operational flow.

Sub-functions within the CEMU layer:
- **Intake** — controlled intake of operational requests within declared scope
- **Governance** — application of Mechanical Ethics policy during execution
- **Fixation** — state fixation at defined checkpoints
- **Sealing** — immutable sealing of execution artifacts
- **Verification** — self-verification of sealed artifacts
- **Closure** — formal closure of the operational capsule
- **Provenance** — chain-of-custody from intake to closure
- **RuntimeGuard** — runtime enforcement boundary (L4 operative level)
- **Enforcement** — fail-closed policy on scope boundary violations

### Layer 4 — Case 003

Case 003 represents the first declared operational deployment within the CEMU capsule framework. It established the scope-bounded execution pattern used in all subsequent AEM / AEM-EVOLVE™ implementations.

### Layer 5 — Publication / Mixed Audience Layer

The publication layer governs how assurance evidence is communicated across technical and non-technical audiences. It mandates claim-language discipline: what can be stated, to whom, and under what boundary conditions.

### Layer 6 — Assurance: SLSA / in-toto / Sigstore

Supply-chain assurance layer applying:
- **SLSA Level 4** — hermetic, reproducible, provenance-tracked builds
- **in-toto** — software supply-chain attestation framework
- **Sigstore** — keyless signing infrastructure

### Layer 7 — RuntimeGuard L4

Operative-level runtime protection:
- Fail-closed enforcement on unauthorized scope expansions
- Hash-verified execution artifacts
- Sealed runtime state at closure checkpoints

### Layer 8 — Regulatory Declared-Scope Layer

Declares the regulatory jurisdictions within which v10.1 claims apply. This layer does **not** assert regulatory approval. It documents the declared mapping surface.

### Layer 9 — Deployment Targets

Declared operational targets at v10.1 factual cutoff:
- `local_controlled` — local controlled environment
- `github_actions_runner` — CI/CD declared scope
- `distributed_target_01` — declared distributed deployment node 1

### Layer 10 — Global Deployment Audit Artifacts

### Layer 11 — Final Readiness Artifacts

Closure artifacts sealing the v10.1 factual state:
