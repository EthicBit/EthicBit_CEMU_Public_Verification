## IX. Clarificación semántica de `block_hash`

Para fines de auditoría e interpretación técnica, la implementación actual de `ClosureAnchor` debe leerse con la siguiente precisión semántica:

En el comportamiento actual del contrato, el valor persistido como `block_hash` corresponde a:

- `blockhash(block.number - 1)`

Esto significa que dicho campo **no representa necesariamente el hash del bloque de la transacción de anchor**. En cambio, representa el hash del **bloque inmediatamente anterior disponible al momento de la ejecución del anchor**.

### Interpretación correcta

El campo actual debe interpretarse como:

- una referencia contextual de cadena capturada en el momento del anchor;
- una referencia al bloque previo;
- y no automáticamente como el hash canónico del bloque en el que la transacción fue minada.

### Significado para auditoría

Esta precisión es importante para evitar lecturas externas incorrectas del receipt y de los artifacts de verificación.

El campo sigue siendo útil como evidencia contextual de cadena, pero su semántica debe entenderse de manera exacta.

### Recomendación de compatibilidad futura

Para versiones futuras, la denominación semánticamente preferible sería alguna de las siguientes:

- `previous_block_hash`
- `anchor_context_block_hash`

Mientras no se implemente ese cambio de nombre, el campo actual `block_hash` deberá interpretarse conforme a la clarificación anterior.
